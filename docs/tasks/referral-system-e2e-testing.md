## Bug 1: Referral Reward Displays Wrong Amount (+15 RCN instead of +25 RCN)

**Severity**: Medium (misleading UI)

**File**: `frontend/src/components/customer/ReferralDashboard.tsx:375`

```tsx
<span className="text-right text-xs font-medium text-green-400">
  {referral.status === "completed"
    ? `+${referral.rewardAmount || 15} RCN`  // <-- Wrong fallback
    : "тАФ"}
</span>
```

The referral object returned by the backend stats API does NOT include a `rewardAmount` field. The backend returns referral records from the `referrals` table which has no reward amount column. So `referral.rewardAmount` is always `undefined`, and the `|| 15` fallback kicks in.

**Actual reward**: 25 RCN for referrer (set in `ReferralService.ts:327`)

**Impact**: Completed referrals show "+15 RCN" instead of "+25 RCN", misleading the referrer about their earnings.

**Fix**:
```tsx
{referral.status === "completed"
  ? "+25 RCN"
  : "тАФ"}
```

Or better тАФ include `rewardAmount` in the backend stats response by joining with the transaction record.

## Bug 2: `totalEarned` Type Mismatch (Number stored as String)

**Severity**: Low (works by accident)

**File**: `frontend/src/components/customer/ReferralDashboard.tsx:51`

```tsx
// State type (line 32):
totalEarned: number;  // <-- Typed as number

// But stored as string (line 51):
totalEarned: resp.data.totalEarned?.toString() || "0",  // <-- String "25" stored in number field
```

The backend returns `totalEarned: 25` (number), but the frontend converts it to string `"25"` before storing in state typed as `number`. This works in the template because JSX concatenation handles both, but it's a type violation that could cause comparison bugs (e.g., `totalEarned > 0` would compare `"25" > 0` тАФ which works in JS but is technically wrong).

**Fix**:
```tsx
totalEarned: resp.data.totalEarned ?? 0,  // Keep as number
```

## Bug 3: Backend Generates Wrong Referral Link URL

**Severity**: Medium (broken links from backend)


The actual registration page is at `/register/customer?ref=CODE`. The backend generates `/register?ref=CODE` which would lead to a 404.

**Impact**: If any system uses the backend-generated `referralLink` (email templates, admin tools, API response), users would get a broken link.

**Fix**:
```typescript
const referralLink = `${process.env.FRONTEND_URL}/register/customer?ref=${referralCode}`;
```

## Bug 4: Column Name Mismatch in Referrals Table

**Severity**: High (potential database errors)

The `referrals` table has inconsistent column naming between schema definition and repository code:

| Location | Column Name Used |
|---|---|
| `setup.ts:172` (CREATE TABLE) | `referee_address` |
| `ReferralRepository.ts:51` (INSERT) | `referred_address` |
| `ReferralRepository.ts:186` (UPDATE in completeReferral) | `referee_address` |
| `ReferralRepository.ts:622` (mapReferralFromDb) | Maps BOTH `row.referred_address` AND `row.referee_address` |

```typescript
// INSERT uses 'referred_address'
INSERT INTO referrals (referral_code, referrer_address, referred_address, ...)

// UPDATE uses 'referee_address'
UPDATE referrals SET referee_address = $1, status = 'completed', ...
```

**Impact**: Depending on whether the actual production column is named `referred_address` or `referee_address`:
- If `referred_address`: INSERT works, but `completeReferral()` UPDATE fails тЖТ referrals can never be marked completed
- If `referee_address`: INSERT fails тЖТ new referrals can never be created
- If BOTH columns exist: Data gets split across two columns, causing lookup failures

**Fix**: Standardize to one column name. Since the CREATE TABLE uses `referee_address`, update the INSERT:
```sql
INSERT INTO referrals (referral_code, referrer_address, referee_address, ...)
```
And update `mapReferralFromDb` to use only `referee_address`:
```typescript
refereeAddress: row.referee_address,  // Remove referredAddress mapping
```

## Bug 5: Dead Referral API Service with Wrong Paths

**Severity**: Low (dead code, no runtime impact)

**File**: `frontend/src/services/api/referral.ts`

The entire referral API service file uses **singular** `/referral/` paths:
```typescript
apiClient.post('/referral/generate', {});       // тЭМ Should be /referrals/generate
apiClient.get('/referral/validate/${code}');      // тЭМ Should be /referrals/validate/${code}
apiClient.get('/referral/stats');                 // тЭМ Should be /referrals/stats
apiClient.get('/referral/leaderboard');           // тЭМ Should be /referrals/leaderboard
apiClient.post('/referral/track-click', {});     // тЭМ Should be /referrals/track-click
apiClient.post('/referral/complete', {});        // тЭМ Should be /referrals/complete
```

Backend is mounted at `/api/referrals` (plural) тАФ `app.ts:322`:
```typescript
this.app.use('/api/referrals', referralRoutes);
```

**Impact**: None currently тАФ the `referralApi` export is NOT imported by any component. The `ReferralDashboard.tsx` uses `apiClient.get("/referrals/stats")` directly (correct path). However, any future developer using the `referralApi` service would get 404 errors.

**Fix**: Either fix the paths to use `/referrals/` (plural) or delete the unused file.

## Bug 6: Dual Referral Code Systems (Potential Confusion)

**Severity**: Low (works in current flow, but confusing architecture)

There are TWO separate referral code storage locations:

| System | Storage | Generated By | Used By |
|---|---|---|---|
| Customer referral code | `customers.referral_code` column | Database DEFAULT/trigger | Dashboard display, registration validation |
| Referral record code | `referrals.referral_code` column | `generate_referral_code()` SQL function | `/api/referrals/generate` endpoint |

**Current flow** (working):
1. Customer's `referral_code` is auto-generated in `customers` table (DB default)
2. Dashboard reads `customerData.referralCode` from customer profile
3. Registration validates against `customers.referral_code` via `getCustomerByReferralCode()`
4. This all stays within the `customers` table тАФ never touches `referrals` table for code lookup

**Dead flow** (unused):
1. `/api/referrals/generate` creates a new code in the `referrals` table
2. This code is DIFFERENT from `customers.referral_code`
3. If shared, registration validation would FAIL because it searches `customers.referral_code`

**Impact**: No current impact since the generate endpoint is unused. But the dual system is confusing for developers.

## Additional Observations (Non-Bugs)

### Referral rewards recorded as `tier_bonus` type
Both referrer and referee rewards are recorded with `type: 'tier_bonus'` (not a dedicated `referral` type). This means in the Transaction History, referral rewards show as "Bonus" rather than "Referral". The `reason` field distinguishes them ("Referral bonus - ...") but users see the generic "Bonus" label.

### No client-side validation of referral code before submit
The registration form accepts any text in the referral code field. Invalid codes only fail on the backend after form submission. Adding a debounced validation on blur (using the `/api/referrals/validate/:code` endpoint) would improve UX.

### `CustomerRegistrationData` interface missing fields
`frontend/src/services/api/customer.ts:13-20` тАФ The TypeScript interface is missing `walletType`, `authMethod`, and `captchaToken` fields that are sent in the actual request payload. Works because JS allows extra properties, but types are inaccurate.
