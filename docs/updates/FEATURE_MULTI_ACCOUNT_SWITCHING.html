Feature Spec

Multi-Account Switching (Facebook-style)

Let a user stay signed into several accounts at once and switch instantly — no logout, no re-verification.

Date:  July 22, 2026 Status:  ◆ Planned — not yet built Area:  Authentication

Summary

Today, switching accounts signs you out and makes you log in again. Facebook doesn't work that way — it keeps every account you've added signed in, and switching between them is a single instant click. This document specifies how to get the same behavior in RepairCoin, why it requires backend work, and where the risk sits.

Why this isn't a UI tweak: the session lives in an httpOnly cookie, which JavaScript deliberately cannot read or swap. Any real switching therefore has to happen on the server. This is an authentication-subsystem change, not a front-end change.

1 Current Behavior vs. Target

StepTodayTarget (Facebook-style)
Add a second account Signs you out of the first First stays signed in
Switch back to the first Full login + email OTP again One click, instant
Accounts remembered Name/avatar only (a shortcut to the login screen) Live sessions

What exists today (shipped): the profile menu remembers accounts you've used and takes you to that account's sign-in with the email pre-filled. It shortens the round trip — it does not remove the logout or the one-time code.

2 Why the server has to be involved

  • The browser session is an httpOnly cookie (auth_token, 15 min + refresh_token, 7 days). Script can't read or rewrite it — that's the point of the flag, and we should not weaken it.
  • Each account is a separate wallet. Signing in requires that account's own proof (an email one-time code, or a wallet signature), so the second sign-in can never be skipped the first time.
  • But it can be skipped on every switch afterwards — if the server remembers that this browser already proved ownership of that account and still holds a valid refresh token for it.
The good news: the foundation already exists. There is a refresh_tokens table and a RefreshTokenRepository (migration 29), with revoke/expiry handling. Multi-session is largely a matter of scoping those tokens to a device and adding a way to activate one.

3 Design

Keep exactly one active session cookie (so every existing route and middleware keeps working untouched), and let the client ask the server to re-point that cookie at another account it has already authenticated on this device. 

// One browser, three remembered accounts, one active session
device_id: "dev_a1b2c3"            // stable per browser, survives logout

refresh_tokens
  ├─ (dev_a1b2c3, 0xAAA…)  valid   // shop  — ACTIVE
  ├─ (dev_a1b2c3, 0xBBB…)  valid   // customer
  └─ (dev_a1b2c3, 0xCCC…)  expired // → needs full sign-in again

POST /auth/switch { address: "0xBBB…" }
  → verify a live refresh token exists for (device_id, 0xBBB…)
  → rotate it, mint a new access token
  → Set-Cookie: auth_token / refresh_token   // now the customer

Backend BE

  1. Migration — add device_id to refresh_tokens, indexed on (device_id, address).
  2. Record it on login — every authenticate endpoint stores the caller's device_id with the issued refresh token.
  3. GET /auth/sessions — returns the accounts with a live, unrevoked token for this device (address, name, role, avatar). This is what the switcher lists.
  4. POST /auth/switch — validates a live token for (device, address), rotates it, and sets the session cookies to that account.
  5. Scope logout — signing out of one account must revoke only that account's token, not the other accounts on the device. "Log out everywhere" stays available as a separate action.

Frontend FE

  1. Device id — generate once, persist locally, survive the logout wipe (the saved-accounts key already does this), and send it on login and switch.
  2. Switcher — the profile menu's "Switch to" list is populated from GET /auth/sessions; picking one calls POST /auth/switch, refetches the profile, and re-renders. No sign-out, no code.
  3. Expired sessions — an account whose token has lapsed falls back to today's behavior: go to sign-in with the email pre-filled.
  4. "Add another account" — signs in a new account without discarding the current one's refresh token, so both are live afterwards.

4 The hard part — wallet vs. session

The endpoints are the straightforward half. The real difficulty is that the front-end auth initializer currently treats the connected wallet as the source of truth: it watches the active wallet and will re-authenticate or sign you out whenever the session and the wallet disagree.

After switching to account B, the connected wallet still holds account A — so that logic will fight the switch and try to pull you back. It has to learn that a session may legitimately differ from the connected wallet.

What makes this tractable: the platform is running in database-only mode, so the wallet isn't required for ordinary actions — it's a login credential, not a per-request dependency. Blockchain-only surfaces (e.g. minting to wallet) should prompt to reconnect the matching wallet on demand.

5 Security requirements

  • Never trust the client's account claim. A switch is only permitted when the server finds a live, unrevoked refresh token for that exact (device, address) pair. The address in the request is a lookup key, never authorization.
  • Keep cookies httpOnly. No design that exposes tokens to JavaScript.
  • Rotate on switch, exactly as the refresh flow already does, so a captured token can't be replayed.
  • Device id is not a secret — it only narrows a lookup. Possessing it must never be sufficient to switch.
  • Rate-limit the switch endpoint, and log switches for audit.
  • Shared devices — "Log out everywhere" must revoke every token for the device.
Failure mode to guard against: session confusion — one account acting with another's privileges. Every switch must fully replace the session identity; nothing account-scoped may be cached across a switch (profile, permissions, tier, shop id, cart).

6 Test plan

  1. Sign into A, add B → both appear in the switcher; A's session survives.
  2. Switch A → B → no logout, no code, correct dashboard and role.
  3. Switch back B → A → instant.
  4. Cross-role: shop ↔ customer, confirming sidebar, permissions and tier all follow the switch.
  5. Expired token → falls back to sign-in with the email pre-filled.
  6. Log out of B → A remains signed in; "log out everywhere" clears both.
  7. Permissions: a staff member's restricted view must not leak across a switch.
  8. Second browser/device → its own session list, unaffected.
  9. Blockchain action after a switch → prompts to reconnect the matching wallet rather than misfiring.

Phases 1 and 2 can ship behind a flag with the switcher hidden, so nothing user-visible changes until phase 3 is proven.

Bottom Line

Facebook-style switching is achievable and the groundwork — persisted, revocable refresh tokens — is already in place. The work is to scope those tokens per device, add two endpoints to list and activate them, and then untangle the front-end assumption that the connected wallet defines who you are. That last piece is the genuine cost, and it is the one to plan for. Until it ships, the interim label change keeps the interface honest about what it actually does.
