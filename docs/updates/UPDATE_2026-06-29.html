Engineering Update

Admin Records Cleanup & Database Hardening

Resolving the root cause of the Sessions crash — and guaranteeing it can't return.

Date:  June 29, 2026 Status:  ✓ Delivered & Verified Environment:  Staging (prod auto-rollout on deploy)

Summary

Today we resolved the underlying cause of a recurring admin-dashboard crash by cleaning up duplicate administrator records in the database, and shipped a permanent safeguard so the issue can never recur. All work was performed safely — with a read-only review first — and has been verified end-to-end.

1 Diagnosed duplicate admin records

The admins table contained duplicate rows for the same team members — the same wallet address had been inserted multiple times over the past months (Jeff, Khalid, and deo each had several entries). This was the underlying cause of the Sessions page "duplicate key" crash and could also skew admin lookups and counts.

2 Reviewed before touching anything

  • Ran a read-only diff first to confirm exactly which rows would be affected — 12 duplicate rows across 3 wallets.
  • Verified there were no foreign-key references pointing at those rows, so removal was safe with zero risk of orphaned data.

3 Cleaned up the duplicates

  • Removed 9 duplicate rows, keeping the oldest, real-name record for each wallet.
  • Correctly preserved the separate, legitimate accounts that only looked similar (a second "Khalid" on a different wallet, and two distinct "Super Administrator" accounts).
  • Confirmed afterward: 0 duplicate wallets remain.

4 Shipped a permanent fix — Migration 189

Created and validated a database migration that does two things, in order:

  • De-duplicates admin records (keeps the oldest row per wallet), and
  • Adds a case-insensitive unique index on the wallet address, so duplicate admins can never be created again.
Self-healing & idempotent. The migration runs automatically on the next deploy — meaning production will be cleaned up and protected with no manual database work required.

5 Verified the safeguard

We tested it by attempting to insert a duplicate admin wallet — the database now correctly rejects it:

ERROR:  duplicate key value violates unique constraint "uq_admins_wallet_lower"
