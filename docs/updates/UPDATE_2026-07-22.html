Engineering Update

Payouts, Marketplace Honesty & a Cleaner Dashboard

Unblocked shop payouts, made the marketplace truthful about who can take bookings, and cut the dashboard navigation roughly in half.

Date:  July 22, 2026 Status:  ✓ Implemented · lint & typecheck clean Focus:  Payments · Navigation · Accounts

Summary

Three threads today. First, shop payouts: the new Stripe Connect flow was failing, which also silently blocked every customer booking — both are now unblocked. Second, the marketplace now tells the truth about which shops can actually accept bookings instead of dead-ending customers. Third, a sizeable navigation cleanup — the shop sidebar went from 17 items to 10, account actions moved into a proper profile menu, and a real account-switching bug was fixed along the way.

1 Stripe Connect — Payouts Unblocked FIXED

"Connect with Stripe" was returning a server error. Stripe Connect is how each shop links its own Stripe account to receive customer payments — but it needs a one-time platform configuration that had never been set up.

  • Diagnosed and documented the missing configuration (the Connect OAuth client id and its redirect URI), added to the environment files with step-by-step setup notes so it can't go missing again.
  • Found the knock-on effect: the same gate blocks bookings. A shop that hasn't connected payouts can't take a booking — there is nowhere for the customer's money to settle — which is why customers were seeing "This shop is not accepting bookings yet." One missing setting was breaking two flows.
  • Made the test shop fully functional so bookings can be exercised end-to-end, including real payment settlement with the platform commission applied.

2 Marketplace — "Preparing" State NEW

Rather than hiding services from shops that haven't finished payout setup, the marketplace keeps them visible but clearly not yet bookable.

  • Service cards show an amber "Preparing" badge, and the Book button is disabled with a tooltip explaining the shop is finishing payout setup.
  • Applied consistently across browse, search, service details, group listings and trending.
  • Safe by default — only services explicitly reported as not-ready are gated; everything else stays bookable.
Why it matters: customers no longer pick a service, fill in a booking, and only then hit a dead-end error. The card tells them up front, and the shop's listings still appear in the marketplace while they finish onboarding.

3 Account Menu on the Profile Chip NEW

The avatar and name in the top-right of both dashboards were decorative. They're now a proper account menu, giving account actions a single consistent home.

  • Shop: Team · Settings · Payouts (Stripe) · Support · Log out.
  • Customer: Settings · FAQ & Help · Log out.
  • Payouts finally has a permanent home. Previously the only way in was a dashboard banner that disappears once a shop is connected — so a shop that dismissed it, or wanted to revisit its payout setup, had nowhere to go.
  • Menu items respect team permissions — staff without team management never see it.

4 Sidebar Cleanup — 17 items → 10 NEW

The shop sidebar had grown to 17 links under 5 collapsible headers, including several duplicates and three headers that wrapped a single link each. It was reorganised in three passes:

  • Removed duplicates — Messages, Settings, Support and Logout all already existed in the header. (Messages is still fully reachable: the header icon previews conversations and opens the full page.)
  • Flattened pointless groups — a section containing one item no longer renders a collapsible header. This is automatic, so it stays true as sections grow or shrink.
  • Merged related destinations behind in-page tabs.

5 Account Switching — Bug Fixed FIXED

Switching accounts silently signed you straight back into the account you had just left. The cause: logging out of the app isn't enough — the wallet layer quietly re-connects on the next page load, and the app treats a connected wallet as a signal to sign in. The session now stays properly signed out so a different account can be used.

The logout routine had also been copy-pasted into three separate places; it's now one shared implementation, which fixed an ordering bug in passing (the server session is now cleared before local data, so a failure can't leave a live session behind).

6 Remembered Accounts NEW

Accounts used on a device are now remembered and listed under "Switch to" in the profile menu, each removable. Picking one takes you straight to that account's sign-in, already greeting them by name with the email pre-filled — instead of logging out, hunting for the login button and retyping it. Only display details are stored; never credentials.

Honest limitation: this shortens the trip, it does not remove the sign-in. Each account is a separate wallet, so it still needs that account's own one-time code. True Facebook-style instant switching requires backend work — specified separately (see below).

7 Next: Facebook-style Account Switching PLANNED

A full specification was written for real multi-account switching — staying signed into several accounts at once and switching instantly, the way Facebook does.

  • Why it isn't a UI tweak: the session is a secure cookie that scripts deliberately cannot read or swap, so switching has to happen on the server.
  • The approach: remember which accounts this browser has already proven it owns, and let the server re-point the active session at any of them on request. The first sign-in per account can never be skipped — every switch afterwards can.
  • The foundation already exists — the platform already stores revocable, expiring session tokens, which is most of what's needed.
  • The real cost is not the new endpoints; it's teaching the app that the signed-in account may legitimately differ from the connected wallet. That's the risky part and is phased accordingly.

Full spec, security requirements, test matrix and phased rollout: docs/updates/FEATURE_MULTI_ACCOUNT_SWITCHING.html

Interim: until it ships, the menu wording should say "log out & sign in as another account" so the interface doesn't imply behavior it doesn't yet have.

8 Testing

Run locally with npm run dev (frontend :3001, backend :4000). A shop must have Stripe payouts connected before its services become bookable — that connection gates both flows below.

A · Shop — create a service

  1. Log in as a shop (active subscription) → Services → Create Service.
  2. Fill in name, price, duration and category; optionally add an image, tags, and configure the Auto Sales & Booking (AI) panel. Save.
  3. Confirm the service appears in the list and its card actions (Edit / Delete / Share QR) behave.
  4. Payout readiness: if the shop isn't connected yet, open Profile menu → Payouts (Stripe) and complete onboarding. The dashboard payout banner should clear once charges are enabled.

B · Customer — book a service

  1. Log in as a customer → open the Marketplace.
  2. Shop not payout-connected: its services show the amber "Preparing" badge with booking disabled — visible, but correctly not bookable.
  3. Shop payout-connected: the service shows Book Now. Pick a date and time and proceed to payment.
  4. Pay with the Stripe test card 4242 4242 4242 4242 (any future expiry, any CVC, any ZIP).
  5. Confirm the booking appears for both sides — the customer's Bookings and the shop's appointments — and that RCN rewards are credited on completion.

C · Navigation & accounts

  1. Sidebar: confirm Bookings, Inventory and Analytics each show their in-page sub-tabs, and that the parent entry stays highlighted while you're on a sub-tab.
  2. Confirm existing links still work directly (appointments, disputes, purchase orders, customers).
  3. Profile menu: open the avatar in the top right on both dashboards; check Settings, Payouts, Support and Team all navigate correctly.
  4. Switch account: confirm you land signed out — not straight back into the same account.
  5. Sign into a second account, then confirm the first appears under "Switch to", and that choosing it opens sign-in already greeting that account.
Test data note: most shops in the dev environment aren't payout-connected yet, so the marketplace will show many "Preparing" cards. Use a connected shop to exercise the full booking-and-payment path.

Bottom Line

Shops can now connect Stripe and receive payouts, and the marketplace is honest about who can take bookings — services from shops still onboarding stay visible but clearly marked "Preparing". The shop dashboard is materially lighter: navigation is down to ten items with account actions consolidated into a profile menu, achieved without touching any page's permissions or access gating. Account switching now actually signs you out, and remembers the accounts you use. The remaining big-ticket item — instant Facebook-style switching between live sessions — is fully specified and ready to schedule, with its riskiest phase identified up front.
