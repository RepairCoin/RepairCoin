Engineering Update

Admin Dashboard — Finishing the Audit Backlog

Every admin tab is now fully functional: accurate numbers, complete management actions, no dead ends.

Date:  July 2, 2026 Status:  ✓ Delivered & Pushed to main Commits:  6 shipped

Summary

Building on yesterday's full audit of the admin dashboard, today we closed out the remaining functional gaps. Admins can now page through the full dispute history, the Subscriptions dashboard shows correct platform-wide numbers, promo codes are fully manageable (create/activate/deactivate/edit/delete), a broken dead-end tab was removed, and the last file with lingering type issues was cleaned up. Every change was type-checked and shipped in small, reviewable commits.

1 Disputes — Pagination FIX

The No-Show Disputes tab fetched a fixed 50 records with no pager, so once there were more than 50 disputes the older ones were unreachable.

  • Added pagination (20 per page) with Prev/Next controls and a “Page X of Y · N total” indicator.
  • The page count is derived from the per-status totals the backend already returns, and the page resets when you switch filter tabs (Pending / Approved / Rejected / All).

2 Subscriptions — Correct Headline Numbers FIX

The headline stats — MRR, Total Revenue, Overdue count, and status counts — were calculated from the current page of results only (max 50). With more than 50 subscriptions, those top-line numbers were simply wrong.

  • They now come from the real platform-wide aggregate endpoint, which computes the totals across all subscriptions in the database.
  • Added a proper overdue count to that aggregate (active subs past their next payment date).
  • Surfaced Billing Method & Billing Reference in the details modal — already fetched from the backend but never displayed, so admins couldn't see how a shop pays.

3 Promo Codes — Full Management NEW

The Promo Codes tab was view-only. Admins could see and search codes but couldn't manage them. It is now a complete management surface:

  • Activate / Deactivate toggle on every code.
  • Edit — a modal to change the name, bonus type/value, start & end dates, and active state (the code and owning shop stay immutable).
  • Delete — with a confirmation dialog that clearly warns it also removes the code's usage history (and shows the use count), and suggests deactivating instead to keep the record.
  • Top Performing Codes — a new ranked section showing the most-used codes with their uses and RCN issued (data the analytics endpoint already returned but was previously discarded).
No accidental data loss: the one destructive action (delete) is guarded by a clear confirmation and a non-destructive alternative (deactivate).

4 Removed a Broken, Dead-End Tab

While reviewing the “Enhanced Metrics” tab, we found it was dead code: it wasn't rendered anywhere and called an API method that doesn't exist, so it could never have worked. Rather than polish a broken, unused file we removed it entirely (~480 lines). Live platform metrics are already fully covered by the Analytics and Advanced Treasury tabs.

5 Codebase Health — Type-Cleanup

Cleaned up the Shop management screen, which carried a cluster of long-standing type issues. The most important was a mismatched value in the confirmation-dialog logic that could have caused the wrong confirmation message to appear. The file now type-checks completely clean (17 issues → 0), with no change to how it behaves for users — just a more reliable, maintainable foundation.
