1 Blockchain Removal — Backend Shipped

Re-architected the entire token system so blockchain is an optional, flag-controlled layer rather than a hard dependency — using a reversible "Provider Pattern" (Strategy B) instead of risky deletion.

Architecture & abstraction

  • Introduced a token-provider abstraction with a database-only provider (now active) and a dormant blockchain provider, selected at runtime by the ENABLE_BLOCKCHAIN_MINTING flag.
  • Routed the core credit / debit / redeem flows through the provider — previously these were broken in database-only mode because they checked an on-chain balance of ~0.

RCG governance token moved off-chain

  • Shop tiers (Standard / Premium / Elite) and RCN purchase pricing now derive from admin-managed database columns instead of reading the blockchain.
  • Converted every consumer: pricing service, shop purchase service, shop RCG info, admin RCG distribution, and the contract reader (made credential-free). No active code path reads RCG on-chain anymore.

Code archive (Phase 3)

  • Converted 14 files that statically imported the smart-contract module to flag-gated dynamic imports — so the contract code only loads when blockchain is enabled.
  • Physically archived the contract modules (token minter, multi-contract minter, blockchain service) into an archive folder — kept for reversibility, not deleted.
  • Created a single lazy admin-contract service as the one entry point for on-chain admin actions, returning safe no-ops when blockchain is off.
  • Removed the last remaining references into the archive folder (even type-only imports) so active code is fully decoupled.

Operational decoupling

  • Decoupled the tier engine from blockchain credentials — the app no longer requires Thirdweb keys to boot in database-only mode.
  • Rolled ENABLE_BLOCKCHAIN_MINTING=false into all in-repo deploy configs (staging auto-deploy, app configs, env templates).

2 Blockchain Removal — Frontend Shipped

Built the mechanism for the UI to detect the platform mode, then audited and gated every blockchain-only surface across customer, shop, and admin areas.

Detection mechanism (new)

  • Added a public runtime-config API endpoint that reports whether blockchain is enabled, mirroring the backend flag as a single source of truth.
  • Built a React config provider + hook consumed app-wide; it "fails closed," so blockchain UI stays hidden if the mode can't be confirmed.

Surfaces audited & gated

  • Customer: "Mint to Wallet" card.
  • Admin: Bulk Mint, Manual Transfer, RCG Transfer page (auto-redirect), and the pending-mints debug page.
  • Shop: RCG Staking (nav item + tab), RCG OTC purchase page (auto-redirect), crypto payment option, and all "Buy RCG" buttons across the dashboard card, onboarding modal & onboarding banner.

A full inventory of the blockchain UI surface area was produced first, so nothing was missed.

Deliberately preserved: wallet login, Stripe-based RCN purchasing, and shop tier/balance display (now database-backed). Per product decision, wallet login stays permanently and is out of scope for removal.

4 Production Bug Fix — Services Marketplace Shipped

Fixed a database error crashing the services marketplace listing (column "s.deleted_at" must appear in the GROUP BY clause).
  • Root cause: the listing query enumerated its grouping columns by hand; when a new column was added to the table, the query broke.
  • Fix: rewrote it to group by primary keys, leveraging PostgreSQL functional dependency — robust and future-proof against new columns.
  • Investigated thoroughly: confirmed soft-deleted services are still correctly excluded and the fix is complete.

Also triaged a "Failed to fetch countries" console warning → confirmed non-fatal (graceful fallback to a built-in country list already in place).

★ Impact & Benefits

  • Lower operating cost & complexity — no gas fees or chain dependency for day-to-day token operations.
  • Faster & more reliable — token earn/redeem now run as instant database transactions, removing on-chain latency and failure modes.
  • Zero-risk reversibility — blockchain can be re-enabled via one flag, with the path proven by an automated test. No "burn the bridges" deletion.
  • Cleaner codebase — contract code archived, dependencies decoupled, no blockchain credentials required to run.
  • Consistent UX — users no longer see actions the platform can't perform in database-only mode.
  • More stable marketplace — fixed a crash affecting the core services listing.
